# 04 Ai Agent 技术栈概览

## 01 到底什么是 AI 应用？

AI 应用，本质上不是“接入了一个大模型 API 的软件”，而是：

```text
用 AI 能力解决某个具体问题的软件产品或功能。
```

也就是说，AI 应用的重点不在于有没有聊天框，也不在于底层用了哪个模型，而在于 AI 是否真正参与了用户任务的完成过程。

一个完整的 AI 应用通常可以理解为：

```text
AI 应用 = 用户任务 + AI 能力 + 数据/上下文 + 工具执行 + 结果反馈
```

### 1. 模型不是应用

GPT、Claude、Gemini 这类大模型只是 AI 能力的来源，类似“发动机”。

但是只有发动机，并不等于有了一辆车。

同样，只是调用一次大模型接口，也不等于做出了一个真正的 AI 应用。模型需要被放到具体业务流程、产品界面、数据系统和工具系统里，才能真正为用户完成任务。

### 2. AI 功能不一定等于 AI 应用

很多传统软件也可以加入 AI 功能，例如：

- 笔记软件里的“总结全文”
- 文档工具里的“改写语气”
- 邮件系统里的“生成回复”
- 表格工具里的“根据数据生成分析”

这些功能当然使用了 AI，但它们更像是传统产品中的 AI 增强能力。

如果一个产品的主要流程还是传统操作，只是在某些地方加了 AI 按钮，那么它更准确地说是“带 AI 功能的软件”。

### 3. 真正的 AI 应用围绕 AI 重新组织流程

更典型的 AI 应用，会把 AI 放在核心工作流中，让用户用更自然的方式完成任务。

例如：

- 用户想写周报，AI 帮他整理信息、生成草稿、调整表达。
- 用户想查公司知识库，AI 理解问题、检索资料、总结答案。
- 用户想处理客服消息，AI 判断用户意图、查询订单、生成回复。
- 用户想订会议室，AI 理解需求、检查日程、调用系统、完成预约。

这些场景里，AI 不只是“生成一段文字”，而是在参与理解、判断、组织信息，甚至调用工具完成动作。

### 4. AI 应用的核心组成

一个 AI 应用通常包含以下几个部分：

| 组成部分 | 作用 |
| --- | --- |
| 用户任务 | 用户真正想完成的事情，例如写作、查询、分析、客服、办公自动化 |
| AI 能力 | 理解语言、生成内容、提取信息、判断意图、规划步骤等 |
| 数据/上下文 | 用户输入、历史记录、业务数据、知识库、数据库内容等 |
| 工具执行 | 调用搜索、数据库、业务接口、文件系统、第三方服务等 |
| 结果反馈 | 把 AI 的输出交给用户确认、修改、继续追问或执行下一步 |

所以，AI 应用不是单纯地“问模型一个问题”，而是把模型能力接入到一个完整的产品闭环中。

### 5. AI 应用和 AI Agent 的关系

AI 应用的范围更大，AI Agent 是其中一种更高级的形态。

普通 AI 应用可能只负责一次性的生成、总结或问答，例如：

```text
用户输入问题 -> 模型生成答案 -> 用户查看结果
```

而 AI Agent 更强调自主规划和工具调用，例如：

```text
用户提出目标 -> AI 拆解步骤 -> 调用工具 -> 检查结果 -> 继续执行 -> 完成任务
```

比如“帮我订一张明天去北京的机票”，这就不只是简单问答。系统需要理解出行需求、查询航班、比较价格、确认用户偏好、调用订票接口，最后返回结果。

这类应用就更接近 AI Agent。

### 6. 判断一个产品是不是 AI 应用

可以用下面几个问题来判断：

- 它是否解决了一个具体用户任务？
- AI 是否参与了理解、生成、判断、决策或执行？
- 它是否结合了业务数据、上下文或工具？
- 用户是否因为 AI 的参与，用更低成本完成了原本复杂的事情？

如果答案是肯定的，那么它就可以被看作 AI 应用。

### 7. 一句话总结

AI 应用就是把大模型、数据、工具和产品流程结合起来，让用户用更自然的方式完成原本复杂、低效或需要专业能力的任务。

它的关键不是“有没有 AI”，而是：

```text
AI 是否真正帮助用户完成了任务。
```

## 02 AI 应用目前的主流形态

目前的 AI 应用不是只有一种样子。根据 AI 在产品里承担的角色不同，大致可以分成几种主流形态。

```text
AI 应用主流形态 = AI 助手 + AI Copilot + AI 工作流 + AI Agent + 垂直行业应用 + 多模态应用
```

这些形态之间不是完全割裂的，很多真实产品会同时包含多种形态。

### 1. AI 助手类应用

AI 助手是最常见、最容易理解的一类 AI 应用。

它的核心交互方式通常是：

```text
用户提问或下达指令 -> AI 理解需求 -> AI 生成回答或内容
```

典型场景包括：

- 问答助手
- 写作助手
- 翻译助手
- 学习助手
- 编程问答助手
- 企业知识库助手

这类应用的特点是以对话为主要入口，用户通过自然语言表达需求，AI 负责理解和生成结果。

例如：

```text
帮我总结这篇文章
帮我写一封请假邮件
解释一下这段代码
根据这份资料生成一个汇报提纲
```

AI 助手类应用的重点是“理解问题”和“生成答案”，通常不一定需要复杂的工具调用。

### 2. AI Copilot 类应用

Copilot 可以理解为“副驾驶”。

它不是单独存在的聊天工具，而是嵌入到某个具体软件或工作场景中，辅助用户完成当前任务。

典型场景包括：

- 编程工具中的代码补全、代码解释、代码重构
- 文档工具中的续写、润色、摘要
- 设计工具中的自动生成布局或文案
- 数据分析工具中的公式生成、图表解释
- CRM 系统中的客户跟进建议

这类应用的特点是：

```text
用户仍然是主操作者，AI 在旁边辅助。
```

例如在代码编辑器中，用户正在写代码，AI 根据当前文件、上下文和光标位置给出补全建议。用户可以接受、修改或拒绝。

AI Copilot 的价值在于提高效率，而不是完全替代用户操作。

### 3. AI 工作流类应用

AI 工作流类应用，会把一个业务流程拆成多个步骤，并在其中某些步骤使用 AI。

它的核心不是自由聊天，而是稳定地完成一套流程。

例如一个“自动生成小红书文案”的工作流可能是：

```text
输入主题 -> 分析目标用户 -> 生成标题 -> 生成正文 -> 生成标签 -> 输出结果
```

再比如一个“简历筛选”工作流可能是：

```text
上传简历 -> 提取候选人信息 -> 匹配岗位要求 -> 给出评分 -> 生成面试问题
```

典型场景包括：

- 内容生产流程
- 客服回复流程
- 简历筛选流程
- 合同审查流程
- 数据报表生成流程
- 营销素材生成流程

这类应用的特点是结果相对可控，流程比较固定，适合企业内部提效。

### 4. AI Agent 类应用

AI Agent 是更进一步的 AI 应用形态。

它不只是生成内容，而是可以围绕一个目标进行拆解、规划、调用工具和持续执行。

普通 AI 应用更像：

```text
你问一句，AI 答一句。
```

AI Agent 更像：

```text
你给一个目标，AI 自己想办法完成多个步骤。
```

例如用户说：

```text
帮我订一张明天上午从上海到北京的机票，价格不要太贵，尽量靠窗。
```

一个 Agent 需要做的事情可能包括：

- 理解用户的出行目标
- 查询航班信息
- 比较时间和价格
- 根据偏好筛选座位
- 向用户确认选择
- 调用订票接口
- 返回订单结果

AI Agent 的关键能力包括：

| 能力 | 说明 |
| --- | --- |
| 任务理解 | 理解用户真正想完成什么 |
| 任务拆解 | 把目标拆成多个可执行步骤 |
| 工具调用 | 调用搜索、数据库、API、浏览器、文件系统等 |
| 状态管理 | 记住当前任务进展和上下文 |
| 结果校验 | 判断执行结果是否满足目标 |
| 多轮反馈 | 根据用户反馈继续调整和执行 |

所以，AI Agent 更接近“能办事的 AI 系统”，而不只是“能聊天的 AI 系统”。

### 5. 垂直行业 AI 应用

垂直行业 AI 应用，是把 AI 能力放到具体行业问题中。

例如：

- 教育：AI 老师、作业批改、个性化学习规划
- 医疗：病历总结、医学问答、辅助诊断
- 法律：合同审查、法律咨询、案例检索
- 金融：研报总结、风险分析、智能投顾
- 电商：智能客服、商品文案、用户评论分析
- 企业服务：知识库问答、会议纪要、销售助手

这类应用的关键不只是大模型能力，而是行业数据、业务规则和场景理解。

同样是问答系统，如果接入的是企业制度、商品信息、合同条款或医学资料，它的产品价值就完全不同。

### 6. 多模态 AI 应用

多模态 AI 应用指的是 AI 不只处理文字，还能处理图片、语音、视频、文件等多种信息。

典型场景包括：

- 图片理解
- 图片生成
- 语音识别
- 语音对话
- 视频总结
- 文档解析
- 截图分析
- 视觉问答

例如：

```text
上传一张截图，让 AI 判断页面问题
上传一份 PDF，让 AI 总结重点
上传一段会议录音，让 AI 生成纪要
上传一张商品图，让 AI 生成营销文案
```

多模态应用的价值在于扩大了 AI 能处理的信息范围，让 AI 从“只会看文字”变成“能看、能听、能读文件”。

### 7. 几种形态的对比

| 形态 | 核心特点 | 典型例子 |
| --- | --- | --- |
| AI 助手 | 以对话问答和内容生成为主 | ChatGPT 类问答、写作助手 |
| AI Copilot | 嵌入具体软件，辅助用户操作 | 编程助手、文档助手、设计助手 |
| AI 工作流 | 按固定流程完成任务 | 自动写文案、简历筛选、合同审查 |
| AI Agent | 能规划步骤并调用工具执行 | 订票 Agent、数据分析 Agent、办公自动化 Agent |
| 垂直行业应用 | 面向特定行业问题 | AI 教育、AI 法律、AI 医疗、AI 客服 |
| 多模态应用 | 处理文字以外的信息 | 图片理解、语音对话、视频总结、文档解析 |

### 8. 学习 AI 应用开发时应该重点关注什么

学习 AI 应用开发，不应该只停留在“怎么调用模型 API”。

更重要的是理解：

- 用户任务是什么
- AI 在流程中承担什么角色
- 需要哪些上下文和业务数据
- 是否需要调用外部工具
- 如何控制输出质量
- 如何处理多轮对话和状态
- 如何把 AI 能力接入真实产品流程

### 9. 一句话总结

AI 应用目前的主流形态，可以从“辅助生成”逐渐发展到“自主执行”：

```text
AI 助手 -> AI Copilot -> AI 工作流 -> AI Agent
```

越往后，AI 参与任务的程度越深，对系统设计、上下文管理、工具调用和结果校验的要求也越高。

## 03 AI 购买机票业务流程梳理

为了理解 AI 应用和 AI Agent 的区别，可以用“AI 帮用户购买机票”这个例子来拆解。

用户的一句话可能是：

```text
帮我订一张明天上午从上海到北京的机票，价格不要太贵，最好靠窗。
```

表面上看，这只是一个简单需求。但如果真的要完成订票，系统背后需要经过一整套业务流程。

### 1. 用户表达目标

第一步是用户用自然语言提出目标。

用户不一定会把所有信息都说完整，例如：

```text
帮我订明天去北京的机票。
```

这句话里缺少很多关键信息：

- 从哪个城市出发？
- 明天上午、下午还是晚上？
- 单程还是往返？
- 几个人出行？
- 是否有价格要求？
- 是否有航空公司偏好？
- 是否需要行李额？
- 是否需要靠窗或靠过道？

传统软件通常要求用户自己填写表单，而 AI 应用可以先理解用户意图，再通过对话补齐信息。

### 2. AI 理解用户意图

AI 首先要判断用户真正想完成的任务是什么。

在这个例子里，用户不是单纯想“查询航班”，而是想“完成订票”。

可以抽象成：

```text
用户目标：购买机票
出发地：上海
目的地：北京
出行时间：明天上午
价格偏好：不要太贵
座位偏好：靠窗
```

这一步主要由 AI 完成，核心能力是自然语言理解和信息提取。

### 3. 系统补全必要信息

如果用户提供的信息不完整，AI 不能直接下单，而是需要追问。

例如：

```text
你是从上海虹桥出发还是上海浦东出发？
请问乘机人是谁？
你希望几点之前到北京？
可以接受中转吗？
```

这里有一个重要原则：

```text
AI 可以推理，但关键业务信息必须确认。
```

例如“明天上午”可以理解为一个时间范围，但具体航班、乘机人、证件信息、支付行为都必须经过用户确认。

### 4. 查询航班信息

信息补齐后，系统需要调用外部工具或业务接口查询航班。

这一步不是大模型自己完成的，而是后台系统调用真实接口完成的。

可能调用的工具包括：

- 航班查询接口
- 价格查询接口
- 舱位库存接口
- 机场信息接口
- 用户常用乘机人信息
- 企业差旅规则系统

流程可以理解为：

```text
AI 整理查询条件 -> 后台调用航班接口 -> 接口返回航班列表 -> AI 参与筛选和解释
```

大模型不能凭空编造航班信息，航班、价格、余票这类数据必须来自真实系统。

### 5. 筛选和推荐方案

接口返回航班列表后，系统需要根据用户偏好筛选。

例如筛选条件可能包括：

- 出发时间是否在上午
- 价格是否相对便宜
- 是否直飞
- 是否有余票
- 到达机场是否合适
- 是否符合靠窗座位偏好
- 是否符合企业差旅标准

AI 可以把多个候选方案整理成用户容易理解的推荐。

例如：

| 方案 | 航班 | 时间 | 价格 | 推荐理由 |
| --- | --- | --- | --- | --- |
| 方案 A | 上海虹桥 -> 北京首都 | 08:30 - 10:45 | 较低 | 时间早，价格合适 |
| 方案 B | 上海浦东 -> 北京大兴 | 09:20 - 11:40 | 中等 | 到达时间合适，座位选择更多 |
| 方案 C | 上海虹桥 -> 北京首都 | 11:00 - 13:15 | 较高 | 航班舒适，但价格偏高 |

推荐时，AI 的作用不是替用户偷偷决定，而是帮助用户理解不同方案的差异。

### 6. 用户确认选择

涉及订单和支付的场景，必须有用户确认环节。

例如 AI 可以回复：

```text
我推荐方案 A：明天 08:30 从上海虹桥出发，10:45 到达北京首都，价格较低，符合上午出发的要求。是否为你预订这一班？
```

用户确认后，系统才可以进入下单流程。

这一环节的重点是：

```text
AI 负责推荐，用户负责最终确认。
```

### 7. 创建订单

用户确认后，后台系统开始创建订单。

创建订单通常需要：

- 航班信息
- 舱位信息
- 乘机人信息
- 证件信息
- 联系方式
- 价格信息
- 退改签规则
- 发票或报销信息

这一步主要由后台系统和业务接口完成。

AI 在这里的作用通常是：

- 整理订单信息
- 向用户解释退改签规则
- 提醒用户核对关键内容
- 在缺少信息时继续追问

### 8. 支付和出票

订单创建后，进入支付和出票环节。

这一步通常不能由 AI 自行完成所有动作，尤其涉及真实付款时，必须经过用户授权。

流程可以是：

```text
创建订单 -> 用户确认支付 -> 调用支付系统 -> 支付成功 -> 调用出票系统 -> 返回出票结果
```

AI 可以做的是帮助用户理解当前状态，例如：

```text
订单已创建，请在 15 分钟内完成支付。
支付成功，正在为你出票。
出票成功，航班信息已发送到你的邮箱。
```

### 9. 异常处理

真实业务中，经常会出现异常情况。

例如：

- 航班售罄
- 价格变化
- 支付失败
- 出票失败
- 用户证件信息错误
- 航班延误或取消
- 企业差旅规则不允许

AI 应用必须能处理这些情况，而不是只会在理想流程里运行。

例如价格变化时：

```text
刚才推荐的航班价格已变化，从 800 元变为 950 元。是否继续预订，还是为你重新筛选更便宜的航班？
```

这类场景里，AI 的价值是把复杂的系统状态翻译成用户能理解的选择。

### 10. 完整流程图

整体业务流程可以总结为：

```text
用户提出目标
-> AI 理解意图
-> 提取出行信息
-> 判断信息是否完整
-> 追问补齐关键信息
-> 调用航班查询接口
-> 获取航班和价格数据
-> 根据偏好筛选方案
-> 向用户推荐并解释
-> 用户确认
-> 创建订单
-> 用户支付
-> 出票
-> 返回结果
-> 处理异常或后续变更
```

### 11. AI、后台、工具分别负责什么

在这个业务里，AI 不是单独完成所有事情，而是和后台系统、外部工具一起协作。

| 角色 | 负责内容 |
| --- | --- |
| 用户 | 提出目标、补充信息、确认方案、完成支付授权 |
| AI | 理解需求、提取信息、追问补齐、筛选解释、生成回复 |
| 后台系统 | 管理业务流程、保存状态、创建订单、控制权限和规则 |
| 外部工具/API | 查询航班、查询价格、检查库存、支付、出票 |
| 数据库 | 保存用户信息、订单信息、对话记录、任务状态 |

这说明 AI 应用开发不是只写 Prompt，也不是只调用模型接口。

真正的 AI 应用开发需要把 AI 能力接到真实业务系统里。

### 12. 从这个例子看 AI Agent 的特点

购买机票这个例子已经非常接近 AI Agent。

因为它不只是回答问题，而是包含：

- 目标理解
- 信息收集
- 任务拆解
- 工具调用
- 多轮对话
- 状态跟踪
- 用户确认
- 异常处理
- 最终执行

也就是说，它已经从“AI 生成答案”变成了“AI 协助完成任务”。

### 13. 一句话总结

AI 购买机票的核心不是让模型编一段航班推荐，而是让 AI 在真实业务流程中完成理解、追问、查询、推荐、确认、下单和异常处理。

可以概括为：

```text
AI 负责理解和决策辅助，后台负责业务流程，工具/API 负责真实执行，用户负责关键确认。
```

## 04 我们 AI 应用学习的重点

学习 AI 应用开发，不能只停留在“会调用大模型 API”。

调用模型只是入门，真正的重点是：

```text
如何把 AI 能力接入真实业务流程，并稳定地帮助用户完成任务。
```

所以，我们学习 AI 应用时，需要同时关注产品、业务、AI 能力和工程实现。

### 1. 先理解用户任务和业务流程

AI 应用的起点不是模型，而是用户任务。

在开发之前，首先要问：

- 用户到底想完成什么？
- 用户现在的流程是什么？
- 哪些步骤低效、重复或依赖经验？
- 哪些环节适合让 AI 参与？
- 哪些环节必须由用户确认？
- 哪些动作必须由后台系统或外部工具执行？

例如“AI 购买机票”这个案例里，用户的目标不是聊天，而是完成订票。AI 只是整个订票流程中的一部分。

所以学习 AI 应用的第一重点是：

```text
先会拆业务流程，再谈模型能力。
```

### 2. 学会设计 AI 在流程中的角色

AI 在不同应用中承担的角色不一样。

它可能是：

- 内容生成者
- 信息总结者
- 问题解答者
- 意图识别者
- 数据提取者
- 方案推荐者
- 工具调用决策者
- 多步骤任务协调者

例如：

| 场景 | AI 的角色 |
| --- | --- |
| 写周报 | 生成和润色内容 |
| 企业知识库问答 | 理解问题并总结资料 |
| 简历筛选 | 提取信息并辅助判断 |
| 机票预订 | 理解需求、追问信息、推荐方案 |
| Agent 办公自动化 | 拆解任务、调用工具、跟踪状态 |

学习时要重点理解：

```text
AI 应该做什么，不应该做什么。
```

例如涉及付款、删除数据、提交订单等高风险动作时，AI 可以建议和准备，但最终确认必须交给用户或业务规则。

### 3. 掌握 Prompt 设计

Prompt 是 AI 应用开发的基础能力。

Prompt 不是简单地写一句“你是一个专业助手”，而是要明确告诉模型：

- 它的角色是什么
- 它要完成什么任务
- 输入数据是什么
- 输出格式是什么
- 有哪些限制条件
- 遇到信息不足时怎么处理
- 哪些事情不能编造

例如：

```text
你是一个机票预订助手。
请从用户输入中提取出发地、目的地、出行日期、时间偏好、价格偏好和座位偏好。
如果关键信息缺失，请只追问缺失的信息，不要直接推荐航班。
输出 JSON 格式。
```

好的 Prompt 能提高模型输出的稳定性，也方便后台程序解析结果。

### 4. 掌握上下文管理

AI 应用不是每次都只处理一句话。

真实应用通常需要记住：

- 用户之前说过什么
- 当前任务进行到哪一步
- 已经收集了哪些信息
- 还缺少哪些信息
- 上一次工具调用的结果是什么
- 用户最终确认了什么

这就是上下文管理。

例如订票场景中，用户第一轮说：

```text
帮我订明天去北京的机票。
```

第二轮补充：

```text
从上海出发，上午的。
```

第三轮又说：

```text
最好便宜一点。
```

系统必须把这些信息合并成完整状态，而不是把每句话孤立处理。

可以理解为：

```text
对话内容 + 用户信息 + 业务状态 + 工具结果 = AI 应用的上下文
```

### 5. 掌握 RAG 和知识库能力

很多 AI 应用不能只依赖模型自带知识。

企业制度、商品信息、课程资料、合同条款、内部文档、订单数据等内容，模型本身并不知道，必须从外部数据源获取。

这时就需要 RAG。

RAG 可以简单理解为：

```text
先从知识库中检索相关资料，再让模型基于资料生成答案。
```

典型场景包括：

- 企业知识库问答
- 智能客服
- 法律条款问答
- 产品说明书问答
- 课程资料问答
- 内部制度查询

学习 AI 应用时，需要重点理解：

- 文档如何切分
- 向量数据库是什么
- 如何根据问题检索相关内容
- 如何把检索结果交给模型
- 如何避免模型脱离资料胡编
- 如何评估回答是否命中原文

### 6. 掌握工具调用和函数调用

AI 应用要完成真实任务，往往不能只靠生成文本。

它还需要调用工具，例如：

- 查询数据库
- 调用业务接口
- 搜索网页
- 读取文件
- 创建订单
- 发送邮件
- 生成报表
- 调用支付系统

这类能力通常通过工具调用或函数调用实现。

可以理解为：

```text
模型负责判断要不要调用工具，程序负责真正执行工具。
```

例如用户说：

```text
帮我查一下明天上午上海到北京的机票。
```

AI 可以判断需要调用“航班查询工具”，但真实航班数据必须由后台接口返回。

学习工具调用时，要重点关注：

- 如何定义工具名称和参数
- 如何让模型选择合适工具
- 如何校验模型生成的参数
- 如何执行工具并返回结果
- 如何把工具结果再次交给模型总结
- 如何处理工具调用失败

### 7. 掌握多轮对话和状态机

复杂 AI 应用通常不是一问一答，而是多轮推进。

例如机票预订流程：

```text
收集需求 -> 补齐信息 -> 查询航班 -> 推荐方案 -> 用户确认 -> 创建订单 -> 支付出票
```

每一步都有不同状态。

如果没有状态管理，AI 很容易出现：

- 重复追问已经提供的信息
- 跳过必要确认
- 不知道当前进行到哪一步
- 在信息不完整时直接执行
- 工具调用顺序混乱

所以复杂 AI 应用通常需要结合状态机或工作流。

可以简单理解为：

```text
状态机负责流程边界，AI 负责理解和生成。
```

### 8. 掌握前端、后台和 AI 的协作方式

AI 应用不是只有 AI 部分。

一个真实 AI 应用通常包含：

| 模块 | 作用 |
| --- | --- |
| 前端 | 展示对话、表单、结果、确认按钮、任务状态 |
| 后台 | 管理业务流程、权限、数据库、接口调用、日志 |
| AI 服务 | 调用模型、处理 Prompt、执行 RAG、工具调用 |
| 数据库 | 保存用户、会话、任务、订单、知识库等数据 |
| 外部 API | 提供真实业务能力，例如查询、支付、出票、搜索 |

前端负责交互体验，后台负责业务可靠性，AI 负责理解和生成。

学习时不能只看 AI，还要理解整个系统如何配合。

### 9. 掌握输出质量控制

AI 输出有不确定性，所以必须做质量控制。

常见问题包括：

- 胡编信息
- 格式不稳定
- 漏掉关键字段
- 错误调用工具
- 逻辑前后矛盾
- 回答太长或太短
- 没有遵守业务规则

常见控制方式包括：

- 明确 Prompt 规则
- 使用结构化输出
- 对模型输出做程序校验
- 关键动作要求用户确认
- 使用 RAG 提供真实资料
- 限制模型不能编造业务数据
- 对工具参数做类型和范围检查
- 记录日志方便排查问题

AI 应用工程化的核心之一，就是让不稳定的模型输出变得尽量可控。

### 10. 掌握异常处理和安全边界

真实业务中一定会出现异常。

例如：

- 模型没有理解用户意图
- 用户输入信息不完整
- 知识库检索不到内容
- 外部接口超时
- 工具调用失败
- 支付失败
- 库存或价格发生变化
- 用户要求执行高风险动作

AI 应用不能只设计理想路径，还要设计失败路径。

尤其是涉及金钱、隐私、订单、医疗、法律、删除数据等场景时，必须有清晰边界。

可以遵循一个原则：

```text
低风险内容可以自动生成，高风险动作必须确认和校验。
```

### 11. 学习路径建议

AI 应用开发可以按下面顺序学习：

```text
模型 API 调用
-> Prompt 设计
-> 结构化输出
-> 多轮对话
-> RAG 知识库
-> 工具调用
-> 工作流和状态管理
-> 前后端集成
-> 日志、评估和质量控制
-> Agent 应用开发
```

这个学习路径的核心是从简单生成，逐步走向真实任务执行。

### 12. 一句话总结

AI 应用学习的重点不是“怎么问模型”，而是“怎么把模型变成产品能力”。

可以概括为：

```text
业务流程是骨架，模型是大脑，数据是上下文，工具是手脚，后台是控制系统，前端是用户入口。
```

真正掌握 AI 应用开发，需要同时理解模型能力、业务流程、系统架构和工程化控制。

## 05 RAG 到底是什么？

RAG 的全称是 Retrieval-Augmented Generation，中文通常叫：

```text
检索增强生成
```

拆开来看：

```text
Retrieval = 检索
Augmented = 增强
Generation = 生成
```

所以 RAG 的核心意思是：

```text
先从外部资料中检索相关内容，再让大模型基于这些资料生成回答。
```

### 1. 为什么需要 RAG？

大模型虽然很聪明，但它有几个天然问题：

- 不知道企业内部资料
- 不知道你的私有文档
- 不知道实时变化的数据
- 可能会胡编答案
- 训练成本高，不能频繁为了新资料重新训练

例如你问模型：

```text
我们公司今年的报销制度是什么？
```

如果公司制度没有提供给模型，模型其实并不知道真实答案。它可能会根据通用经验编一个看起来合理的回答，但这个回答不一定正确。

RAG 的作用就是把真实资料先找出来，再交给模型回答。

可以理解为：

```text
没有 RAG：模型凭记忆回答
有 RAG：模型看着资料回答
```

### 2. RAG 不是训练模型

很多初学者容易把 RAG 理解成“把文档喂给模型训练”。

这个理解不准确。

RAG 通常不是重新训练大模型，而是在用户提问时，临时从知识库里查出相关资料，然后把资料放进 Prompt 里，让模型基于这些资料生成答案。

也就是说：

```text
训练模型 = 改变模型本身的参数
RAG = 不改模型，只把相关资料作为上下文提供给模型
```

这也是 RAG 在企业应用里很常见的原因：成本低、更新快、可控性更强。

### 3. 一个简单例子

假设我们有一份公司请假制度文档，里面写着：

```text
员工请年假需要提前 3 个工作日提交申请。
病假需要提供医院证明。
婚假为 10 天，需要提交结婚证复印件。
```

用户问：

```text
我想请婚假，可以请几天？需要什么材料？
```

RAG 系统会先从知识库中检索到和“婚假”相关的内容，然后把这段内容交给模型。

模型再回答：

```text
根据公司请假制度，婚假为 10 天，需要提交结婚证复印件。
```

这时模型不是凭空回答，而是基于检索到的资料回答。

### 4. RAG 的基本流程

一个典型 RAG 流程可以分成两部分：

- 知识库准备阶段
- 用户提问回答阶段

知识库准备阶段：

```text
准备文档
-> 文档切分
-> 生成向量
-> 存入向量数据库
```

用户提问回答阶段：

```text
用户提问
-> 将问题转成向量
-> 在向量数据库中检索相关片段
-> 把检索结果放进 Prompt
-> 大模型基于资料生成答案
-> 返回给用户
```

完整理解就是：

```text
文档先入库，问题来时先检索，再生成答案。
```

### 5. 文档切分是什么？

真实文档往往很长，不能直接全部塞给模型。

所以需要先把文档切成较小的片段，这个过程叫文档切分，也叫 chunking。

例如一份员工手册可以切成：

- 入职流程片段
- 请假制度片段
- 报销制度片段
- 绩效考核片段
- 离职流程片段

用户问请假问题时，只需要检索请假制度相关片段，而不需要把整本员工手册都交给模型。

文档切分的质量会直接影响 RAG 效果。

如果切得太碎，信息不完整；如果切得太大，检索不精准。

### 6. 向量和向量数据库是什么？

为了让系统知道“用户问题”和“文档片段”是否相关，需要把文字转换成向量。

向量可以简单理解为：

```text
一段文字在语义空间中的数字表示。
```

比如：

```text
“请婚假需要什么材料？”
```

和：

```text
“婚假需要提交结婚证复印件。”
```

这两句话字面上不完全一样，但语义很接近。转换成向量后，它们在语义空间里的距离就会比较近。

向量数据库的作用，就是存储这些文档向量，并在用户提问时快速找出最相关的内容。

常见向量数据库包括：

- Chroma
- Milvus
- Pinecone
- Weaviate
- FAISS
- pgvector

### 7. RAG 在 AI 应用中的典型场景

RAG 非常适合“答案必须基于指定资料”的场景。

典型应用包括：

- 企业知识库问答
- 智能客服
- 课程资料问答
- 产品说明书问答
- 法律合同问答
- 医疗资料检索辅助
- 内部制度查询
- 技术文档助手
- 论文阅读助手

这些场景的共同特点是：

```text
答案不能只靠模型通用知识，必须参考具体资料。
```

### 8. RAG 和普通问答的区别

| 对比项 | 普通大模型问答 | RAG 问答 |
| --- | --- | --- |
| 知识来源 | 主要来自模型训练知识 | 来自外部知识库和检索结果 |
| 私有数据 | 默认不知道 | 可以接入企业或个人资料 |
| 实时更新 | 更新困难 | 文档入库后即可更新 |
| 可控性 | 容易胡编 | 可以要求基于资料回答 |
| 可追溯性 | 通常无法说明来源 | 可以返回引用片段或来源 |
| 成本 | 不需要知识库，但准确性受限 | 需要搭建检索系统，但更适合业务 |

RAG 的关键价值不是让模型变得更聪明，而是让模型回答时有资料可查。

### 9. RAG 能解决什么问题？

RAG 主要解决三类问题。

第一，解决模型不知道私有知识的问题。

例如企业内部制度、客户资料、产品手册、课程文档，这些内容模型默认不知道。

第二，减少模型胡编。

通过要求模型基于检索资料回答，可以降低它凭空生成错误内容的概率。

第三，让知识更新更方便。

如果公司制度更新了，不需要重新训练模型，只需要更新知识库中的文档。

### 10. RAG 不能解决什么问题？

RAG 很有用，但不是万能的。

它不能保证所有回答都一定正确。

常见问题包括：

- 文档本身不完整
- 文档切分不合理
- 检索结果不相关
- 召回了错误资料
- 模型没有严格依据资料回答
- 用户问题太模糊
- 多份资料之间存在冲突

所以做 RAG 应用时，不能只搭一个向量库就结束。

还需要考虑：

- 如何清洗文档
- 如何设计切分策略
- 如何提高检索准确率
- 如何让模型引用来源
- 如何在找不到资料时拒绝回答
- 如何评估回答质量

### 11. RAG 和 Agent 的关系

RAG 可以看作 Agent 或 AI 应用中的一种能力。

RAG 主要解决：

```text
去哪里查资料，查到资料后如何回答。
```

Agent 更进一步，解决：

```text
为了完成目标，需要做哪些步骤，需要调用哪些工具。
```

例如一个企业办公 Agent 可能会同时使用 RAG 和工具调用：

```text
用户问制度 -> Agent 调用 RAG 查知识库 -> 返回制度解释
用户要请假 -> Agent 调用请假系统 -> 创建请假申请
```

所以 RAG 不是 Agent，但 RAG 经常是 Agent 的重要组成部分。

### 12. 一句话总结

RAG 就是让大模型在回答之前先查资料。

可以概括为：

```text
RAG = 检索相关资料 + 基于资料生成答案
```

它的核心价值是把大模型和外部知识库连接起来，让 AI 应用可以回答企业资料、私有文档和实时更新的数据问题。

## 06 LLM 选型与对比

LLM 选型，不是简单地问“哪个模型最强”。

真正做 AI 应用时，更重要的问题是：

```text
在当前业务场景下，哪个模型最合适？
```

因为不同模型在推理能力、速度、成本、上下文长度、多模态、工具调用、稳定性和部署方式上都不一样。

### 1. 不要只追求最强模型

最强模型不一定是最适合业务的模型。

例如：

- 客服分类任务可能更适合便宜、快速的小模型。
- 合同审查可能需要推理能力更强、上下文更长的大模型。
- 语音对话需要低延迟模型。
- 企业私有化部署可能需要开源模型。
- Agent 工具调用需要模型有较好的规划和函数调用能力。

所以 LLM 选型的核心不是“谁最强”，而是：

```text
能力够用，成本可控，速度可接受，结果可评估。
```

### 2. LLM 选型的核心维度

选模型时，可以从下面几个维度判断。

| 维度 | 关注点 |
| --- | --- |
| 推理能力 | 能不能处理复杂问题、多步骤任务、逻辑判断 |
| 生成质量 | 文案、总结、改写、问答是否自然准确 |
| 代码能力 | 是否适合代码生成、代码解释、调试、重构 |
| 工具调用 | 是否擅长函数调用、参数生成、任务规划 |
| 上下文长度 | 能不能处理长文档、长对话、大量代码 |
| 多模态能力 | 是否支持图片、音频、视频、文件理解 |
| 结构化输出 | 是否能稳定输出 JSON、表格、固定格式 |
| 速度延迟 | 响应是否足够快，能否用于实时交互 |
| 成本 | 输入输出 token 成本是否适合业务量 |
| 稳定性 | 输出是否稳定，是否容易胡编或跑偏 |
| 生态工具 | SDK、文档、社区、平台能力是否成熟 |
| 合规部署 | 是否支持私有化、本地部署、企业安全要求 |

### 3. 主流模型家族的基本定位

不同厂商的模型会不断更新，具体型号和价格要以官方文档为准。

但从产品定位上，可以粗略理解为：

| 模型家族 | 常见特点 | 适合场景 |
| --- | --- | --- |
| OpenAI GPT 系列 | 综合能力强，工具调用和生态完善 | 通用 AI 应用、Agent、代码、结构化输出、多模态 |
| Anthropic Claude 系列 | 长文本、代码、复杂推理和 Agent 表现突出 | 长文档分析、代码库理解、企业知识工作、复杂 Agent |
| Google Gemini 系列 | 多模态、长上下文和 Google 生态整合能力强 | 多模态应用、长文档、音视频、Google 生态应用 |
| DeepSeek 系列 | 性价比高，推理和代码能力有竞争力 | 成本敏感场景、中文场景、代码和推理任务 |
| 开源模型 | 可私有化、可微调、可控性强 | 私有部署、数据安全要求高、定制化场景 |

截至 2026-06-23，官方资料中可以看到一些典型定位：

- OpenAI 将 GPT-5.5 定位为复杂推理和编码的旗舰模型，同时提供更低延迟、低成本的小型号。
- Anthropic 当前有 Fable/Mythos、Opus、Sonnet、Haiku 等不同层级，其中 Opus 偏复杂推理和长周期 Agent 工作，Sonnet 偏速度与能力平衡，Haiku 偏高吞吐和成本敏感任务。
- Google Gemini 提供 Pro、Flash、Flash-Lite、Live 等不同定位，分别面向复杂任务、低延迟高吞吐、预算敏感和实时音频场景。
- DeepSeek API 中也区分普通对话和推理能力，并持续迭代模型命名和兼容方式。

这说明主流厂商基本都会提供类似分层：

```text
旗舰模型：能力最强，成本较高
中型模型：能力和成本平衡
小型模型：速度快，价格低，适合高并发
```

### 4. 按任务类型选择模型

不同任务对模型能力的要求不一样。

| 任务类型 | 推荐模型类型 |
| --- | --- |
| 简单分类、标签提取 | 小模型或低成本模型 |
| 普通问答、摘要、改写 | 中型通用模型 |
| 复杂推理、数学、规划 | 强推理模型 |
| 代码生成和代码审查 | 代码能力强的旗舰或中高端模型 |
| 长文档分析 | 长上下文模型或结合 RAG |
| 企业知识库问答 | 中型模型 + RAG + 引用来源 |
| 实时语音对话 | 低延迟实时模型 |
| 图像理解、多模态问答 | 多模态模型 |
| Agent 工具调用 | 工具调用稳定、规划能力强的模型 |
| 大批量低风险任务 | 小模型、批处理或缓存优化 |

简单来说：

```text
任务越复杂，越需要强模型；任务越高频，越要关注成本和速度。
```

### 5. 按业务阶段选择模型

不同开发阶段，选型策略也不一样。

早期验证阶段：

```text
优先选能力强的模型，把产品流程先跑通。
```

原因是早期最大风险不是成本，而是不确定这个需求能不能被 AI 做好。

中期优化阶段：

```text
开始评估哪些任务可以换成更便宜、更快的模型。
```

例如主流程用强模型，简单分类、格式化、标题生成用小模型。

生产稳定阶段：

```text
建立模型评估、降级策略、缓存策略和成本监控。
```

这时不要只依赖一个模型，最好有备用方案。

### 6. 强模型和小模型如何搭配

真实 AI 应用里，不一定所有任务都用同一个模型。

可以采用模型分层：

```text
强模型：负责复杂推理、任务规划、关键决策
中型模型：负责普通问答、总结、改写
小模型：负责分类、抽取、格式转换、低风险批量任务
Embedding 模型：负责向量化和语义检索
Rerank 模型：负责检索结果重排序
```

例如一个智能客服系统可以这样设计：

| 环节 | 可用模型类型 |
| --- | --- |
| 判断用户意图 | 小模型 |
| 检索知识库 | Embedding 模型 |
| 重排检索结果 | Rerank 模型 |
| 生成正式回复 | 中型模型 |
| 复杂投诉处理 | 强模型 |
| 高风险退款操作 | 强模型 + 规则校验 + 人工确认 |

这样既能控制成本，也能保证关键环节质量。

### 7. 云端模型和开源模型怎么选

LLM 大致可以分成两类：

- 云端闭源模型
- 开源或可本地部署模型

云端模型的优点：

- 能力强
- 接入快
- 运维成本低
- 工具生态成熟
- 多模态和 Agent 能力更新快

云端模型的缺点：

- 数据需要发到第三方服务
- 成本随调用量增长
- 受 API 限制和服务稳定性影响
- 模型升级可能影响输出

开源或本地模型的优点：

- 数据可控
- 可私有化部署
- 可微调和深度定制
- 长期成本可能更可控

开源或本地模型的缺点：

- 部署和运维成本高
- 推理硬件成本高
- 模型效果可能需要调优
- 工具调用和多模态能力可能不如云端模型成熟

可以简单判断：

```text
追求最快上线和最强能力：优先云端模型
追求数据安全和私有化控制：考虑开源或本地模型
```

### 8. 不同场景的选型建议

如果是个人项目或课程学习：

```text
优先选择文档完善、SDK 成熟、调试方便的云端模型。
```

如果是企业内部知识库：

```text
中型模型 + RAG + 权限控制 + 引用来源。
```

如果是客服系统：

```text
小模型做意图识别，中型模型生成回复，复杂问题升级强模型或人工。
```

如果是代码 Agent：

```text
选择代码能力、工具调用能力、长上下文能力都比较强的模型。
```

如果是实时语音助手：

```text
优先考虑低延迟、流式输出和实时音频能力。
```

如果是高并发内容生成：

```text
优先考虑成本、速度、批处理和缓存。
```

如果是金融、医疗、法律等高风险场景：

```text
不要只看模型能力，还要重点设计审核、溯源、权限、合规和人工确认。
```

### 9. 选型时最容易犯的错误

常见错误包括：

- 只看排行榜，不看业务场景
- 只看模型能力，不看成本
- 只看单次效果，不做批量评估
- 用强模型处理所有任务
- 没有备用模型和降级策略
- 不记录模型输入输出日志
- 没有评估 Prompt 修改后的影响
- 忽略模型版本升级带来的变化
- 把 RAG、工具调用、业务规则的问题都归因给模型

很多时候，效果不好不一定是模型不行，而是：

```text
Prompt 不清楚、上下文不完整、检索不准确、工具设计不合理、业务流程没拆清楚。
```

### 10. 如何做模型评估

不要只靠主观感觉选模型。

更可靠的方法是准备一批真实测试用例。

测试集可以包括：

- 常见问题
- 边界问题
- 模糊表达
- 长上下文问题
- 复杂推理问题
- 工具调用问题
- 结构化输出问题
- 容易胡编的问题
- 高风险场景问题

然后对不同模型进行对比：

| 评估项 | 说明 |
| --- | --- |
| 准确率 | 回答是否正确 |
| 稳定性 | 多次调用结果是否一致 |
| 格式合规 | 是否按要求输出 JSON 或固定结构 |
| 幻觉率 | 是否编造不存在的信息 |
| 工具调用成功率 | 是否能正确选择工具并生成参数 |
| 延迟 | 用户等待时间是否可接受 |
| 成本 | 单次调用和总调用量成本 |
| 用户满意度 | 真实用户是否觉得好用 |

模型选型不是一次性决定，而是持续评估和调整。

### 11. 推荐的选型流程

可以按下面流程选模型：

```text
明确业务场景
-> 拆分任务类型
-> 选择 2-3 个候选模型
-> 准备真实测试集
-> 对比效果、速度和成本
-> 先用强模型跑通流程
-> 再逐步替换低成本模型
-> 建立监控、评估和降级方案
```

这个流程比单纯问“哪个模型最好”更可靠。

### 12. 一句话总结

LLM 选型的本质是工程权衡。

可以概括为：

```text
先看任务，再看能力；先跑通流程，再优化成本；先做评估，再做替换。
```

真正成熟的 AI 应用，通常不是只依赖一个最强模型，而是根据不同任务组合使用多个模型，并持续做评估和成本优化。

## 07 LLM 在 AI 应用中目前充当什么角色？

LLM 是 Large Language Model，也就是大语言模型。

在 AI 应用里，LLM 很重要，但它不是整个应用本身。

更准确地说：

```text
LLM 是 AI 应用里的智能能力核心，但不是完整业务系统。
```

它更像 AI 应用中的“理解、生成、判断和调度引擎”。

### 1. LLM 不是应用本身

很多初学者容易把 LLM 和 AI 应用混在一起。

例如：

```text
调用 GPT API = 做了一个 AI 应用
```

这个理解不准确。

LLM 只是提供智能能力，真正的 AI 应用还需要：

- 前端界面
- 后台服务
- 数据库
- 用户系统
- 权限系统
- 业务流程
- 工具/API
- 日志监控
- 结果评估
- 安全边界

所以 LLM 更像发动机，AI 应用才是整辆车。

### 2. LLM 充当自然语言理解模块

LLM 最基础的能力，是理解用户用自然语言表达的需求。

例如用户说：

```text
帮我订一张明天上午从上海到北京的机票，价格不要太贵。
```

LLM 可以从中理解：

```text
任务：订机票
出发地：上海
目的地：北京
时间：明天上午
偏好：价格不要太贵
```

这类能力在 AI 应用里非常关键，因为它让用户不必按照固定表单填写信息，而是可以用自然语言表达目标。

### 3. LLM 充当信息提取和结构化模块

AI 应用经常需要把用户输入或文档内容转成结构化数据。

例如把一句话转成 JSON：

```text
帮我订明天上午上海到北京的机票。
```

可以提取成：

```json
{
  "task": "book_flight",
  "from": "上海",
  "to": "北京",
  "date": "明天",
  "time_preference": "上午"
}
```

这时 LLM 的角色不是聊天，而是把模糊的人类语言转成程序能处理的数据。

常见场景包括：

- 从简历中提取候选人信息
- 从合同中提取关键条款
- 从用户消息中识别意图
- 从客服对话中提取投诉原因
- 从会议纪要中提取待办事项

### 4. LLM 充当内容生成模块

内容生成仍然是 LLM 最常见的用途之一。

例如：

- 写文章
- 写邮件
- 写周报
- 写营销文案
- 生成客服回复
- 生成代码
- 生成摘要
- 生成学习计划

这类场景里，LLM 的作用是根据输入和上下文生成自然语言内容。

但在真实业务中，生成内容通常还需要受到约束：

- 不能编造事实
- 不能违反业务规则
- 要符合输出格式
- 要符合品牌语气
- 要能被用户修改或确认

所以内容生成不是让模型自由发挥，而是让模型在规则和上下文内生成。

### 5. LLM 充当推理和判断模块

LLM 还能做一定程度的推理、分类和判断。

例如：

- 判断用户意图
- 判断问题是否需要升级人工
- 判断合同条款是否有风险
- 判断简历和岗位是否匹配
- 判断下一步应该追问什么
- 判断工具调用结果是否满足需求

例如客服场景中，用户说：

```text
我已经等了三天还没收到货，能不能赶紧处理？
```

LLM 可以判断：

```text
意图：物流问题
情绪：不满
优先级：较高
下一步：查询订单物流状态
```

这说明 LLM 不只是生成文字，还能参与业务流程中的判断。

### 6. LLM 充当工具调用决策模块

目前很多 AI 应用都不是只让模型回答，而是让模型判断是否需要调用工具。

工具可以是：

- 搜索工具
- 数据库查询
- 订单系统
- 支付系统
- 邮件系统
- 日历系统
- 文件系统
- 企业内部 API

例如用户问：

```text
帮我查一下订单 12345 到哪了。
```

LLM 不应该凭空回答，而是应该判断：

```text
需要调用订单查询工具。
```

然后由后台真正执行查询：

```text
LLM 生成工具参数 -> 后台调用订单接口 -> 接口返回结果 -> LLM 总结给用户
```

这里的关键是：

```text
LLM 负责判断和生成参数，真实执行由程序和工具完成。
```

OpenAI、Anthropic、Google 等主流平台现在都把工具调用作为构建 AI 应用和 Agent 的重要能力。

### 7. LLM 充当上下文整合模块

AI 应用的输入往往不只有用户当前一句话。

还可能包括：

- 历史对话
- 用户资料
- 业务数据
- RAG 检索结果
- 工具调用结果
- 当前任务状态
- 系统规则

LLM 的一个重要角色，是把这些上下文整合起来，然后生成合适的下一步。

例如：

```text
用户当前问题 + 历史对话 + 知识库资料 + 订单状态 -> LLM 生成回答
```

所以 LLM 不只是处理一句 prompt，而是在上下文中理解当前任务。

这也是为什么 AI 应用开发越来越强调：

```text
Context Engineering，上下文工程。
```

### 8. LLM 在 Agent 中充当规划和调度核心

在 AI Agent 中，LLM 的角色会更进一步。

它不只是回答问题，而是参与：

- 理解目标
- 拆解任务
- 选择工具
- 生成工具参数
- 观察工具结果
- 判断是否继续执行
- 总结最终结果

一个简化的 Agent 循环可以理解为：

```text
用户目标
-> LLM 理解目标
-> LLM 决定下一步
-> 调用工具
-> 获取结果
-> LLM 判断是否完成
-> 继续执行或返回答案
```

Anthropic 把这类系统中的基础构件称为 augmented LLM，也就是被检索、工具和记忆增强过的 LLM。

也就是说，Agent 不是只有一个模型，而是：

```text
LLM + 工具 + 上下文 + 状态 + 规则 + 执行循环
```

### 9. LLM 不负责什么？

为了避免误解，也要知道 LLM 不应该负责什么。

LLM 不应该直接负责：

- 保存核心业务数据
- 直接修改数据库
- 直接执行支付
- 直接删除重要数据
- 直接决定高风险审批
- 凭空生成实时数据
- 替代权限系统
- 替代业务规则系统
- 替代日志和审计系统

这些事情应该由后台系统、数据库、权限系统、业务规则和人工确认来控制。

可以简单理解：

```text
LLM 可以建议、判断和生成，但关键执行必须由可控系统完成。
```

### 10. LLM 和传统程序如何配合？

AI 应用不是用 LLM 替代所有代码。

更常见的方式是：

```text
传统程序负责确定性逻辑
LLM 负责非确定性理解和生成
```

例如：

| 模块 | 更适合谁负责 |
| --- | --- |
| 用户登录 | 传统程序 |
| 权限校验 | 传统程序 |
| 数据库读写 | 传统程序 |
| 支付下单 | 传统程序 |
| 自然语言理解 | LLM |
| 内容生成 | LLM |
| 意图判断 | LLM |
| 多工具选择 | LLM + 程序校验 |
| 高风险确认 | 传统程序 + 用户确认 |

所以 AI 应用开发不是“把所有逻辑交给模型”，而是设计好模型和程序的分工。

### 11. 一个完整位置图

可以用下面这个结构理解 LLM 在 AI 应用中的位置：

```text
用户
-> 前端界面
-> 后台服务/编排层
-> LLM 理解、生成、判断、规划
-> RAG/数据库/工具/API
-> 后台校验和状态管理
-> 返回结果给用户
```

LLM 位于系统的智能决策层，但它需要被产品流程、业务规则和工程系统包裹起来。

### 12. 一句话总结

LLM 在 AI 应用中目前主要充当“智能引擎”。

可以概括为：

```text
LLM 负责理解语言、生成内容、提取信息、辅助判断、选择工具和规划步骤；
后台系统负责数据、规则、权限、状态和真实执行。
```

真正成熟的 AI 应用，不是让 LLM 单独完成一切，而是让 LLM 在合适的位置发挥智能能力。

## 08 Prompt 提示词的 5 大要素

Prompt 是提示词，也就是我们给大模型的指令。

很多人一开始写 Prompt，只是简单地问一句：

```text
帮我写一篇文章。
```

这种写法模型也能回答，但输出结果往往不稳定。因为模型不知道你的身份、目标、背景、限制条件和输出格式。

一个更好的 Prompt，通常要包含 5 个核心要素：

```text
角色 + 任务 + 上下文 + 约束 + 输出格式
```

### 1. 角色：告诉模型它是谁

角色决定模型应该用什么身份、视角和专业能力来回答。

例如：

```text
你是一名资深产品经理。
你是一名 Python 教学老师。
你是一名企业客服主管。
你是一名合同审查律师。
你是一名 AI 应用架构师。
```

角色不是为了让模型“表演”，而是为了限定回答风格和专业视角。

比如同一个问题：

```text
帮我分析这个功能。
```

如果角色是产品经理，它会关注用户需求、使用流程和商业价值。

如果角色是后端工程师，它会关注接口、数据库、权限和性能。

所以角色的作用是：

```text
让模型站在正确的位置思考问题。
```

### 2. 任务：告诉模型要做什么

任务是 Prompt 中最核心的部分。

任务要尽量明确，不要太模糊。

模糊任务：

```text
帮我优化一下。
```

更明确的任务：

```text
请帮我把下面这段课程笔记改写得更适合初学者理解，保留核心概念，语言更口语化。
```

任务可以是：

- 总结
- 翻译
- 改写
- 分类
- 提取信息
- 生成文案
- 分析问题
- 设计方案
- 生成代码
- 检查错误

一个好的任务描述，应该让模型知道：

```text
要处理什么内容，要产出什么结果。
```

### 3. 上下文：告诉模型背景信息

上下文是模型理解任务所需要的背景资料。

例如：

- 用户是谁
- 使用场景是什么
- 当前业务流程是什么
- 已有信息是什么
- 参考资料是什么
- 对话历史是什么
- 目标读者是谁
- 当前限制是什么

如果没有上下文，模型只能根据通用经验回答。

例如：

```text
帮我写一段介绍。
```

这个请求太空了。模型不知道介绍什么、给谁看、用在什么场景。

加上上下文后：

```text
我在写一门面向前端工程师的 AI 应用开发课程。
请帮我写一段“什么是 RAG”的介绍，要求适合没有机器学习背景的人理解。
```

这样模型就更容易生成符合需求的内容。

可以理解为：

```text
上下文越清楚，模型越不容易乱猜。
```

### 4. 约束：告诉模型不能怎么做

约束是为了控制模型输出边界。

常见约束包括：

- 字数限制
- 语言风格
- 不要编造
- 不要使用术语
- 必须基于资料回答
- 信息不足时要追问
- 不要输出无关解释
- 不要改变原意
- 不要执行高风险动作

例如：

```text
请用中文回答。
控制在 300 字以内。
不要使用太多专业术语。
如果资料中没有答案，请直接说“资料中未提到”，不要编造。
```

在 AI 应用中，约束非常重要。

因为业务系统通常不希望模型自由发挥，而是希望它在规则内完成任务。

可以简单理解：

```text
任务告诉模型要做什么，约束告诉模型不要做什么。
```

### 5. 输出格式：告诉模型结果长什么样

输出格式决定模型返回结果是否方便用户阅读，或者是否方便程序解析。

如果是给人看的内容，可以要求：

- 列表
- 表格
- 分段说明
- 简短总结
- 标题 + 正文

如果是给程序处理，可以要求：

- JSON
- 固定字段
- 数组结构
- Markdown 表格

例如：

```text
请用下面格式输出：

标题：
核心观点：
详细说明：
示例：
一句话总结：
```

如果是结构化输出：

```json
{
  "intent": "book_flight",
  "from": "上海",
  "to": "北京",
  "date": "明天",
  "missing_fields": []
}
```

在 AI 应用开发中，输出格式尤其重要，因为后台程序经常需要读取模型结果。

例如：

```text
模型输出 JSON -> 后台解析字段 -> 判断是否调用工具 -> 执行业务流程
```

### 6. 一个完整 Prompt 示例

把 5 个要素组合起来，可以得到一个更完整的 Prompt。

示例：

```text
角色：
你是一名 AI 应用课程讲师，擅长用通俗语言解释技术概念。

任务：
请解释“RAG 到底是什么”。

上下文：
这是一门面向前端工程师和 AI 应用初学者的课程。学习者懂一些 Web 开发，但不一定懂机器学习。

约束：
不要使用过多数学术语。
不要把 RAG 解释成训练模型。
请强调 RAG 是“先检索资料，再基于资料生成答案”。

输出格式：
请按下面结构输出：
1. 简单定义
2. 为什么需要它
3. 一个生活化例子
4. 一句话总结
```

这个 Prompt 比“解释一下 RAG”更稳定，因为它把模型需要的信息都说清楚了。

### 7. 5 大要素对照表

| 要素 | 作用 | 示例 |
| --- | --- | --- |
| 角色 | 限定模型身份和视角 | 你是一名 AI 应用架构师 |
| 任务 | 明确要完成什么 | 请解释 RAG 的工作流程 |
| 上下文 | 提供背景和资料 | 面向没有机器学习背景的前端工程师 |
| 约束 | 控制边界和规则 | 不要编造，不超过 300 字 |
| 输出格式 | 规定结果结构 | 用 Markdown 表格输出 |

### 8. 常见错误

写 Prompt 时常见的问题包括：

- 任务太模糊
- 没有提供背景
- 没有说明目标用户
- 没有限制输出范围
- 没有要求输出格式
- 一次要求模型做太多事
- 让模型基于不存在的信息回答
- 希望模型猜测业务规则

例如：

```text
帮我写一个方案。
```

这句话就太模糊了。模型不知道是什么方案、给谁看、目标是什么、格式是什么。

更好的写法是：

```text
你是一名 AI 产品经理。
请为一个企业知识库问答系统设计 MVP 方案。
背景是公司有大量内部文档，员工经常找不到制度和流程。
要求方案控制在 800 字以内，不要涉及复杂模型训练。
请按“目标、核心功能、技术方案、风险点、下一步计划”输出。
```

### 9. 在 AI 应用中如何使用 Prompt

在真实 AI 应用中，Prompt 通常不是用户直接写的。

更常见的方式是：

```text
用户输入 -> 后台组装 Prompt -> 调用模型 -> 解析结果 -> 返回用户或调用工具
```

后台会把以下内容组合起来：

- 系统角色
- 用户输入
- 历史对话
- 业务规则
- RAG 检索结果
- 工具说明
- 输出格式要求

所以 AI 应用开发中的 Prompt 不是一句话，而是一套可维护的指令模板。

### 10. 一句话总结

Prompt 的 5 大要素是：

```text
角色、任务、上下文、约束、输出格式。
```

好的 Prompt 不是把话说得更玄，而是把需求说得更清楚，让模型知道：

```text
以什么身份，在什么背景下，完成什么任务，遵守什么规则，输出什么格式。
```

## 09 什么是 MCP？

MCP 的全称是 Model Context Protocol，中文可以理解为：

```text
模型上下文协议
```

它是一个开放协议，用来标准化 AI 应用和外部工具、数据源之间的连接方式。

简单说：

```text
MCP = 让 AI 应用用统一方式连接工具和数据的协议。
```

### 1. 为什么需要 MCP？

在没有 MCP 之前，如果一个 AI 应用想连接不同工具，通常要分别写很多适配代码。

例如：

- 连接 GitHub 要写一套接口
- 连接数据库要写一套接口
- 连接文件系统要写一套接口
- 连接浏览器要写一套接口
- 连接企业内部系统还要再写一套接口

每个 AI 应用都要重复做这些集成，非常麻烦。

可以理解为：

```text
没有 MCP：每个 AI 应用都要单独对接每个工具
有 MCP：工具按统一协议暴露能力，AI 应用按统一协议调用
```

这有点像 USB-C 的作用。

不同设备只要遵循同一个接口标准，就可以更容易连接在一起。

MCP 对 AI 应用来说，就是一种“连接外部能力的标准接口”。

### 2. MCP 解决的核心问题

LLM 本身只能根据上下文生成内容。

它默认不能直接：

- 读取你的本地文件
- 查询数据库
- 操作浏览器
- 查看 GitHub 仓库
- 查询企业订单
- 调用内部业务系统
- 获取实时工具结果

如果想让 AI Agent 真正办事，就必须让它能连接外部系统。

MCP 解决的就是这个连接问题。

可以概括为：

```text
LLM 负责理解和决策，MCP 负责把外部工具和数据接进来。
```

### 3. MCP 的基本架构

MCP 中常见几个角色：

| 角色　　 | 作用　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　　 |
| ----------| ------------------------------------------------------------------------|
| Host　　 | 用户直接使用的 AI 应用，例如 Claude Desktop、Cursor、Codex、IDE 插件等 |
| Client　 | Host 里面负责和 MCP Server 通信的客户端　　　　　　　　　　　　　　　　|
| Server　 | 按 MCP 协议暴露工具、资源或提示词的服务　　　　　　　　　　　　　　　　|
| Tool　　 | 可以被 AI 调用的动作，例如查数据库、读文件、发请求　　　　　　　　　　 |
| Resource | 可以提供给 AI 读取的数据，例如文件、文档、记录　　　　　　　　　　　　 |
| Prompt　 | 预定义的提示词模板或工作流入口　　　　　　　　　　　　　　　　　　　　 |

可以用下面的结构理解：

```text
用户
-> AI Host
-> MCP Client
-> MCP Server
-> 工具 / 数据源 / 外部系统
```

例如：

```text
Claude Desktop
-> MCP Client
-> 文件系统 MCP Server
-> 读取本地文件
```

或者：

```text
AI 编程助手
-> MCP Client
-> GitHub MCP Server
-> 查询 issue、PR、代码仓库
```

### 4. MCP Server 是什么？

MCP Server 是按 MCP 协议提供能力的服务。

它可以把某个外部系统包装成 AI 能理解和调用的能力。

例如：

- 文件系统 MCP Server：提供读写文件能力
- GitHub MCP Server：提供仓库、Issue、PR 操作能力
- 数据库 MCP Server：提供查询数据库能力
- 浏览器 MCP Server：提供打开网页、点击、截图能力
- 企业内部 MCP Server：提供订单、客户、审批等业务能力

MCP Server 的重点不是“变成一个聊天机器人”，而是：

```text
把某个系统的能力标准化暴露给 AI 应用。
```

### 5. Tools、Resources、Prompts 分别是什么？

MCP 里经常会看到三个核心能力：Tools、Resources、Prompts。

#### Tools

Tools 是可以被模型调用的动作。

例如：

```text
查询订单
读取文件
创建 issue
搜索文档
调用接口
发送邮件
```

Tool 通常会有明确的输入参数和输出结果。

例如：

```json
{
  "tool": "get_order_status",
  "arguments": {
    "order_id": "12345"
  }
}
```

#### Resources

Resources 是可以提供给模型读取的数据。

例如：

```text
一个文件
一篇文档
一条数据库记录
一个项目配置
一份 API 文档
```

Resources 更像“可读取的上下文资料”。

#### Prompts

Prompts 是预定义的提示词模板。

例如：

```text
代码审查模板
生成周报模板
分析错误日志模板
创建 PR 描述模板
```

Prompts 可以让某些常用任务被标准化复用。

### 6. MCP 和 Function Calling 的关系

Function Calling 是模型调用函数或工具的一种能力。

MCP 则更像是工具和数据的连接协议。

可以这样理解：

```text
Function Calling：模型如何决定调用哪个函数
MCP：外部工具如何以统一方式暴露给 AI 应用
```

二者并不冲突，反而经常一起使用。

例如：

```text
模型判断需要查订单
-> 通过工具调用生成参数
-> AI 应用通过 MCP Server 调用订单系统
-> 返回结果给模型总结
```

### 7. MCP 和 RAG 的关系

RAG 主要解决：

```text
从知识库中检索资料，再让模型基于资料回答。
```

MCP 解决：

```text
AI 应用如何连接外部数据源和工具。
```

所以 MCP 可以用来接入 RAG 系统，但 MCP 不等于 RAG。

例如：

```text
用户问公司制度
-> AI 通过 MCP 调用知识库检索工具
-> 返回相关制度片段
-> 模型基于片段生成回答
```

在这个流程里，RAG 是检索增强生成的方法，MCP 是连接知识库工具的协议。

### 8. MCP 和 Agent 的关系

Agent 要真正完成任务，通常需要调用工具。

MCP 正好提供了一种标准方式，让 Agent 能连接外部工具。

可以理解为：

```text
Agent = 会规划和执行任务的 AI 系统
MCP = Agent 连接工具和数据的标准接口
```

例如一个代码 Agent 可能通过 MCP：

- 读取项目文件
- 搜索代码
- 查看 GitHub Issue
- 运行测试
- 创建 PR

一个企业办公 Agent 可能通过 MCP：

- 查询员工信息
- 查询审批流程
- 创建请假申请
- 发送通知
- 查询知识库资料

所以 MCP 对 Agent 很重要，因为它让 Agent 不只是“会说”，还能更标准地“连接外部能力”。

### 9. MCP 的优势

MCP 的主要优势包括：

- 标准化工具接入方式
- 降低 AI 应用和外部系统的集成成本
- MCP Server 可以被多个 AI 应用复用
- 工具能力可以独立开发和维护
- 更适合构建可扩展的 Agent 生态
- 可以把企业内部系统封装成 AI 可调用能力

可以简单理解：

```text
以前是每个应用重复造连接器，现在可以把连接能力做成 MCP Server 复用。
```

### 10. MCP 不是什么？

为了避免误解，也要知道 MCP 不是什么。

MCP 不是大模型。

它不会让模型本身变聪明。

MCP 也不是一个完整 Agent。

它只负责提供工具和数据的连接标准。

MCP 也不是业务系统本身。

订单系统、数据库、文件系统、GitHub 这些真实系统仍然存在，MCP 只是把它们包装成 AI 应用可以访问的接口。

可以概括为：

```text
MCP 不是大脑，而是连接外部世界的接口。
```

### 11. 一个简单例子

假设用户在 AI 编程助手里说：

```text
帮我看看这个项目里登录功能是怎么实现的。
```

如果没有工具，模型只能根据用户复制的代码回答。

如果接入文件系统 MCP Server，流程可能是：

```text
用户提出问题
-> AI 判断需要读取项目文件
-> 调用文件搜索工具
-> 找到 login、auth 相关文件
-> 读取文件内容
-> LLM 分析代码
-> 返回解释
```

这里 MCP 的价值在于让 AI 可以标准化地访问项目文件，而不是让用户手动复制所有代码。

### 12. 学习 MCP 时应该关注什么

学习 MCP 时，可以重点关注：

- Host、Client、Server 的关系
- Tool、Resource、Prompt 的区别
- MCP Server 如何暴露能力
- AI 应用如何发现和调用工具
- 工具参数如何定义
- 工具结果如何返回给模型
- 权限和安全边界如何控制
- 如何把业务系统包装成 MCP Server

尤其要记住：

```text
MCP 的核心价值是连接，不是替代模型，也不是替代业务系统。
```

### 13. 一句话总结

MCP 是 AI 应用连接外部工具和数据源的开放协议。

可以概括为：

```text
MCP = AI 应用和外部世界之间的标准连接层。
```

它让 Agent 可以用更统一的方式读取数据、调用工具、访问系统，从而从“只能聊天”走向“可以连接真实业务能力”。

## 10 LangChain 的作用？

课程目录里的 `LangChian` 应该是 `LangChain` 的拼写误差。

LangChain 是一个用于构建 LLM 应用和 Agent 应用的开发框架。

简单说：

```text
LangChain = 帮开发者把模型、Prompt、工具、RAG、Agent 和工程流程组织起来的框架。
```

它不是大模型本身，也不会让模型自动变聪明。

它的作用更像是 AI 应用开发里的“工程脚手架”。

### 1. 为什么需要 LangChain？

如果只是简单调用一次大模型 API，直接写代码就可以。

例如：

```text
用户输入 -> 调用模型 -> 返回结果
```

但真实 AI 应用往往不只是一次模型调用。

它可能需要：

- 组装 Prompt
- 接入不同模型
- 解析模型输出
- 接入知识库
- 调用工具
- 管理多轮对话
- 编排多个步骤
- 记录运行过程
- 调试和评估效果

如果所有东西都自己从零写，会比较复杂。

LangChain 的作用就是把这些常见能力封装成可复用的组件，让开发者更快搭建 AI 应用。

### 2. LangChain 解决的核心问题

LangChain 主要解决的是 LLM 应用开发中的工程组织问题。

可以概括为：

```text
模型会生成内容，但应用需要流程、工具、状态、数据和工程管理。
```

LangChain 帮你把这些东西串起来。

例如一个企业知识库问答应用，可能需要：

```text
用户提问
-> 改写问题
-> 检索知识库
-> 取回相关文档
-> 组装 Prompt
-> 调用模型
-> 生成答案
-> 返回引用来源
```

LangChain 可以帮助组织这些步骤。

### 3. LangChain 常见能力

LangChain 常见能力包括：

| 能力 | 作用 |
| --- | --- |
| Model 接入 | 统一调用不同模型供应商 |
| Prompt 模板 | 管理和复用提示词 |
| Output Parser | 把模型输出解析成 JSON、列表、对象等结构 |
| RAG 组件 | 接入文档加载、切分、向量库、检索器 |
| Tool 调用 | 把外部函数或 API 包装成模型可调用工具 |
| Agent | 让模型根据目标选择工具并执行多步骤任务 |
| Memory/State | 管理对话历史和任务状态 |
| Integration | 对接数据库、向量库、搜索、文件系统等生态 |
| Tracing | 记录链路，方便调试和排查问题 |

这些能力本质上都是为了让 LLM 更容易进入真实应用流程。

### 4. LangChain 和直接调用模型 API 的区别

直接调用模型 API 更像：

```text
自己写所有流程，只在某一步调用模型。
```

LangChain 更像：

```text
使用一套框架来组织模型调用、工具调用、RAG 和 Agent 流程。
```

对比一下：

| 方式 | 特点 |
| --- | --- |
| 直接调用模型 API | 简单、灵活、依赖少，适合小功能或高度定制 |
| 使用 LangChain | 组件丰富、生态成熟，适合快速搭建复杂 LLM 应用 |

如果只是写一个“总结文本”的小功能，直接调用 API 就够了。

如果要做知识库问答、Agent、工具调用、多步骤流程，LangChain 会更有帮助。

### 5. LangChain 和 RAG 的关系

RAG 是一种方法：

```text
先检索资料，再基于资料生成答案。
```

LangChain 是一个框架，可以帮你实现 RAG。

例如 LangChain 可以提供或集成：

- 文档加载器
- 文档切分器
- Embedding 模型
- 向量数据库
- 检索器
- Prompt 模板
- 模型调用
- 输出解析

所以：

```text
RAG 是应用方案，LangChain 是实现 RAG 的工具之一。
```

不用 LangChain 也可以做 RAG，只是你需要自己组织更多工程代码。

### 6. LangChain 和 Agent 的关系

LangChain 也可以用来构建 Agent。

Agent 场景中，模型不只是生成答案，还要判断下一步做什么。

例如：

```text
用户目标
-> 模型判断需要调用哪个工具
-> 执行工具
-> 观察结果
-> 继续调用工具或返回答案
```

LangChain 可以帮你：

- 定义工具
- 把工具暴露给模型
- 创建 Agent
- 管理多轮执行过程
- 解析工具调用结果

不过更复杂、需要状态机和可控流程的 Agent，现在通常会使用 LangGraph。

### 7. LangChain、LangGraph、LangSmith 的区别

现在 LangChain 生态里经常一起出现三个名字：

| 名称 | 作用 |
| --- | --- |
| LangChain | 快速构建 LLM 应用和 Agent 的基础框架 |
| LangGraph | 编排更复杂、状态化、长流程的 Agent 工作流 |
| LangSmith | 调试、追踪、评估和监控 LLM/Agent 应用 |

可以这样理解：

```text
LangChain：搭应用
LangGraph：编排复杂 Agent 流程
LangSmith：观察、调试、评估和监控
```

如果做简单 LLM 应用，LangChain 就够用。

如果做复杂 Agent，例如多步骤、有状态、有人类确认、可能循环执行的任务，就更适合使用 LangGraph。

如果应用要上线生产环境，就需要 LangSmith 这类可观测和评估工具。

### 8. LangChain 和 MCP 的关系

MCP 是连接外部工具和数据的开放协议。

LangChain 是构建 LLM 应用的框架。

二者不是一类东西。

可以这样理解：

```text
MCP：工具和数据如何标准化暴露出来
LangChain：AI 应用如何组织模型、工具、RAG 和 Agent 流程
```

LangChain 也可以通过 MCP adapter 使用 MCP Server 暴露的工具。

例如：

```text
MCP Server 暴露 GitHub 工具
-> LangChain Agent 加载这些工具
-> 模型根据任务调用 GitHub 工具
```

### 9. LangChain 适合什么场景？

LangChain 比较适合：

- 快速开发 AI 原型
- 构建知识库问答
- 构建 RAG 应用
- 构建工具调用应用
- 构建 Agent 应用
- 对接多个模型供应商
- 集成向量数据库、搜索、文档加载器
- 做多步骤 LLM 流程编排

例如：

```text
企业知识库问答
合同分析助手
客服自动回复
数据分析 Agent
文档总结工具
代码问答助手
```

### 10. 什么时候不一定需要 LangChain？

不是所有 AI 应用都必须用 LangChain。

下面这些场景可以直接调用模型 API：

- 只做一次简单问答
- 只做文本总结或改写
- 业务逻辑很简单
- 不需要 RAG
- 不需要 Agent
- 不需要复杂工具调用
- 团队希望完全控制底层代码

有时候直接写代码反而更清晰。

可以简单判断：

```text
流程简单：直接调用模型 API
流程复杂：考虑 LangChain / LangGraph
```

### 11. LangChain 的价值和局限

LangChain 的价值：

- 提供成熟组件，减少重复造轮子
- 方便快速搭建原型
- 集成大量模型、向量库和工具生态
- 适合学习和理解 LLM 应用常见模式
- 可配合 LangSmith 做调试和评估

LangChain 的局限：

- 抽象层较多，初学时可能觉得复杂
- 简单项目可能显得过重
- 框架版本迭代快，需要关注文档变化
- 复杂业务仍然需要自己设计流程、权限和异常处理

所以不能把 LangChain 理解成“用了就自动做好 AI 应用”。

它只是帮你组织工程，但业务设计、Prompt、RAG 质量、工具安全和系统架构仍然要自己负责。

### 12. 一句话总结

LangChain 的作用是降低 LLM 应用和 Agent 应用的工程开发成本。

可以概括为：

```text
LangChain 不是模型，而是把模型、Prompt、工具、知识库、Agent 流程和调试能力组织起来的开发框架。
```

它适合用来快速搭建复杂 AI 应用，但是否使用它，要看项目复杂度和团队对工程控制的要求。
